# 物流追踪系统 - 详细下一步计划

## 阶段一：环境准备（1-2天）

### Day 1：账号注册与开通

**上午（2小时）**
- [ ] 注册微信小程序账号（企业/个体户）
  - 准备：营业执照照片、法人身份证
  - 访问：https://mp.weixin.qq.com/
  - 完成邮箱验证、信息登记

- [ ] 开通微信小程序云开发
  - 登录小程序后台
  - 点击"云开发"按钮
  - 记录云环境ID（格式如：xxx-xxx-xxx）

**下午（2小时）**
- [ ] 申请Kimi API Key
  - 访问：https://platform.moonshot.cn/
  - 注册账号，实名认证
  - 创建API Key，保存好（只显示一次）

- [ ] 下载安装微信开发者工具
  - 访问：https://developers.weixin.qq.com/miniprogram/dev/devtools/download.html
  - 安装并登录

### Day 2：项目配置

**上午（2小时）**
- [ ] 修改云环境ID
  ```
  文件：前端小程序/app.js
  修改：env: '你的云环境ID'
  ```

- [ ] 导入项目到开发者工具
  - 选择"导入项目"
  - 项目目录：选择"前端小程序"文件夹
  - AppID：填写你的小程序AppID
  - 勾选"使用云开发"

**下午（3小时）**
- [ ] 部署云函数
  ```
  顺序：
  1. 右键 cloudfunctions/config → "创建并部署"
  2. 右键 cloudfunctions/ai-parse → "创建并部署：云端安装依赖"
  3. 右键 cloudfunctions/shipment-create → "创建并部署：云端安装依赖"
  4. 右键 cloudfunctions/shipment-query → "创建并部署：云端安装依赖"
  5. 右键 cloudfunctions/shipment-update → "创建并部署：云端安装依赖"
  ```

- [ ] 设置云函数环境变量
  ```
  路径：云开发控制台 → 云函数 → 设置
  添加：KIMI_API_KEY=你的API密钥
  ```

---

## 阶段二：数据库搭建（1天）

### Day 3：数据库创建

**上午（2小时）**
- [ ] 创建集合
  ```
  在云开发控制台 → 数据库：
  1. 点击"添加集合" → 输入"shipments"
  2. 点击"添加集合" → 输入"users"
  3. 点击"添加集合" → 输入"operation_logs"
  ```

- [ ] 创建索引
  ```
  shipments集合：
  - managerPhone_1_createdAt_-1
    - managerPhone: 升序
    - createdAt: 降序
  
  - shareToken_1
    - shareToken: 升序

  users集合：
  - phone_1
    - phone: 升序

  operation_logs集合：
  - shipmentId_1_timestamp_-1
    - shipmentId: 升序
    - timestamp: 降序
  ```

**下午（2小时）**
- [ ] 申请微信订阅消息模板
  ```
  路径：微信公众平台 → 功能 → 订阅消息
  1. 点击"添加" → 搜索"物流"
  2. 找到"物流状态更新提醒"模板
  3. 点击"申请"，填写使用场景
  4. 等待审核通过（通常即时通过）
  5. 复制模板ID到 config/index.js
  ```

## 阶段三：功能测试（2-3天）

### Day 4：AI识别测试

**测试1：正常单据识别**
- [ ] 准备3-5张不同格式的进仓通知单
- [ ] 在"建单"页面上传
- [ ] 验证字段提取是否准确：
  - 进仓编号
  - 件数、毛重、体积
  - 工厂名称
  - 仓库信息

**测试2：手写检测**
- [ ] 找一张单据，手写修改几个数字
- [ ] 上传测试
- [ ] 验证：
  - 是否显示"检测到手写涂改"警告
  - 手写字段是否黄色高亮
  - 是否需点击确认才能提交

**测试3：边界情况**
- [ ] 光线暗的图片
- [ ] 模糊的图片
- [ ] 倾斜的图片
- [ ] 记录AI识别准确率

### Day 5：建单流程测试

**测试1：标准流程**
- [ ] 上传单据 → AI识别 → 人工校对 → 选择路线 → 创建成功
- [ ] 验证生成的单号格式：26MHxxxxxx
- [ ] 验证分享链接是否可用
- [ ] 验证数据库是否正确保存

**测试2：不同路线**
- [ ] 测试"义乌→宁波（经义乌仓）"
- [ ] 测试"义乌→宁波（直达）"
- [ ] 测试"工厂直送宁波"
- [ ] 验证各路线节点是否正确

**测试3：异常处理**
- [ ] 网络中断时创建
- [ ] 必填字段为空时提交
- [ ] 手写字段未确认时提交

### Day 6：查询与更新测试

**测试1：查询功能**
- [ ] 按单号查询
- [ ] 按手机号查询（需先绑定）
- [ ] 通过分享链接访问
- [ ] 验证各端显示是否正常

**测试2：节点更新**

## 阶段四：内部试运行（3-5天）

### Day 7-9：内部测试

**测试目标**：处理10-20票真实货物

**每日流程**：
1. 早上：用系统创建当天要处理的货物
2. 白天：实际运输过程中更新节点
3. 晚上：检查数据是否准确，记录问题

**重点关注**：
- [ ] 单号是否连续无重复
- [ ] 时间节点是否准确记录
- [ ] 分享链接发给客户后反馈
- [ ] 手机端查看体验

### Day 10：问题修复与优化

**根据测试反馈修改**：
- [ ] 修复发现的bug
- [ ] 优化AI提示词（如果识别率低）
- [ ] 调整通知文案
- [ ] 优化页面加载速度

## 风险预案

| 风险 | 预案 |
|------|------|
| AI识别率低 | 切换到人工录入模式，继续优化Prompt |
| 云开发费用超支 | 设置预算告警，评估迁移方案 |
| 用户不绑定小程序 | 保留"复制文案"兜底方案 |
| 系统崩溃 | 保留传统电话/Excel备用流程 |
