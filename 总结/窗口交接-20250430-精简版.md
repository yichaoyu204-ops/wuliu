## 2. 关键决策记录

| 决策 | 原因 |
|------|------|
| 物流节点与 OA 流程分离 | 仓管员只管入库/出库，管理员只管确认价格，互不干扰 |
| 仓管员分类基于 timeline 节点 | oaStatus 变化慢，timeline 节点能实时反映货物物理位置 |
| 现结在实测定价时收款 | 仓管员测完数据当场收款，避免后续催款 |
| 确认出库需先实测定价 | 防止未定价就出库，导致费用纠纷 |
| 删除 queue 节点 | 实际业务中没有"排队"这个独立环节 |

## 3. 首次错误与纠正

**错误1**：最初把"所有运单"功能给所有角色可见
- 我的理解：首页功能入口所有人都能看到
- 为什么错：忽略了用户说"只有管理员界面才有全部运单"
- 纠正：用 wx:if 控制，只在 isAdmin 时显示

**错误2**：计费逻辑未先转吨
- 我的理解：直接用公斤算 ratio
- 为什么错：漏掉了"除1000转吨"这个关键步骤
- 纠正：weightTons = weightKg / 1000，ratio = volume / weightTons

**错误3**：云函数部署后仍报"网络错误"
- 我的理解：部署了就生效
- 为什么错：notification.js 引用了不存在的 ./aiParser 模块
- 纠正：把 truncatePushSummary 内联到 notification.js 中

**错误4**：权限不足（白名单为空）
- 我的理解：空数组应该放通
- 为什么错：代码中没有实现"空数组放通"逻辑
- 纠正：在 isWarehouseOrAdmin 和 isAdmin 中添加 whitelistEmpty 判断

## 4. 根因分析

- **需求理解**：用户说"仓库现结"时，我最初没意识到这和月结是两条不同流程
- **上下文遗漏**：漏掉了 CLAUDE.md 中"切换窗口必须先确认"的指令
- **执行偏差**：云函数修改后忘了提醒用户部署，导致反复报错
- **过度简化**：把物流节点推进和 OA 流程混为一谈，后来才分离

## 5. 用户真正想要的效果

仓管员、管理员、业务员各自看到自己该看的、做自己该做的，物流状态实时同步给所有人，但权限严格分离。

## 6. 下次避免规则

- 涉及多角色时，先画流程图再写代码
- 云函数修改后**立刻**提醒部署，不要等用户报错
- 提到"所有角色"时，逐一确认每个角色的差异点
- 计费逻辑必须和用户逐字核对公式
